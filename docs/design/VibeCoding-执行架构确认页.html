✅ 最终决策（2026-07-18，Perry 拍板）：vibe coding 仅限对话创建 Skill / Workflow / Chatflow / Agent 资产，不提供通用软件开发能力。因产出物为结构化配置而非可执行代码：无需沙盒隔离（以 Schema 校验 / 权限边界 / 草稿-审核状态机 / 审计 四道防线替代）；无独立执行服务，回归 Next.js 单体；服务器不部署 Claude Code client；Workflow 的 Code/Template 任意代码节点首版禁用。Copilot 功能并入切片 1/3/4（任务 4.1.6 / 4.3.3 / 4.4.5）。若未来开放通用软件开发能力，重启下方 Q2/Q3/Q5 的沙盒设计。

一、现状盘点：哪些已清晰、哪些是空白

✅ 已清晰（可直接进入配置开发）
  • 产品内普通对话/Agent 的 LLM 接入：Kimi 2.5（Moonshot API，OpenAI 兼容），Agent 级 config.model 调配，Key 服务端管理，call_logs 计量——ADR-003 闭环
  • Claude Code 与 Kimi 的适配通路：Moonshot 提供 Anthropic 兼容端点 api.moonshot.cn/anthropic，两个环境变量即接通
  • 员工个人终端用 Claude Code：属工具订阅问题（GLM/Kimi Coding Plan），不涉及服务器开发
❌ 未清晰（阻塞开发的空白，即本页要确认的）
  • PRD 空白：v1.06 十大模块中没有「vibe coding 工作台」——它的用户故事、界面形态、验收标准均未定义
  • 执行引擎未选型：服务器上跑的是 Claude Code headless、Claude Agent SDK、还是自研 agent 循环？各方案的许可、成本、维护差异大
  • 沙盒隔离方案未定：多用户并发执行 AI 生成的代码，隔离怎么做、资源怎么限、服务器扛几个并发
  • 架构冲突：ADR-001 定的是 Next.js 单体，而沙盒/进程管理需要常驻执行服务——需要一次明确的架构豁免决策
  • 信息缺口：服务器（43.173.99.218）的 CPU/内存/磁盘规格未知，直接决定并发容量设计

二、7 个决策点

Q1 · 产品形态：vibe coding 在 AIPaddle 里是什么？

这决定 PRD 增章的写法和 UI 形态（编码需要文件树/终端输出/diff 视图，与聊天窗差异大）。

Q2 · 执行引擎：服务器上跑什么？

三条路线都能接 Kimi 2.5（Anthropic 兼容端点），差别在能力完整度、可控性与许可合规。

Q3 · 沙盒隔离：AI 生成的代码在哪执行？

全员使用 = 不可信代码多租户并发执行，这是安全红线（等级与租户隔离同级）。

Q4 · 工作区与产物：代码存哪、怎么带走？

决定用户"昨天写的东西今天还在吗"和交付方式。

Q5 · 沙盒网络策略：能联网装包吗？

vibe coding 几乎必然要 npm/pip 装依赖，但全开放等于给 AI 代码开了外联通道。

Q6 · 架构豁免：执行服务与 Next.js 单体的关系？

容器编排、长会话、流式输出不适合塞进 Next.js API Routes，需对 ADR-001"单体"做一次有记录的豁免。

Q7 · 排期：插入哪个批次？

当前批次 2-4 是地基→Agent→知识库。vibe coding 是重功能，插队会推迟核心闭环。
