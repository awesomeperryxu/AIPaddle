# AIPaddle 测试策略

> 版本 v1.0 · 2026-07-18 · 配套 `ROADMAP.md` 与 `.github/ISSUE_TEMPLATE/task.md`
> 原则：**测试跟着任务粒度走**——每层测试都有明确的"何时跑、谁来跑、跑什么"，不存在"最后统一测试"这回事。

---

## 总览：六层测试 × 五个时机

| 层 | 测什么 | 工具 | 何时跑 |
|----|--------|------|--------|
| L1 静态检查 | 类型错误、代码规范、能否构建 | TypeScript + ESLint（`pnpm check`） | **每个任务**完成时；CI 每次 push |
| L2 单元测试 | 纯逻辑：状态机、权限矩阵、工具函数、组件关键交互 | Vitest + React Testing Library | **每个任务**（涉及逻辑就写）；CI 每次 push |
| L3 集成/API 测试 | API 路由：入参校验、权限、租户隔离、数据落库 | Vitest + Supabase 本地栈 | **每个 API 任务**；CI 每次 push |
| L4 E2E 端到端 | 真实浏览器里走通用户路径 | Playwright | **每个切片**完成时；CI 合并 main 前 |
| L5 专项测试 | 租户隔离（P0）、RAG 质量、性能、安全 | 见下文 | 对应切片完成时 |
| L6 人工测试 | 手动验收（每任务）+ 真实用户测试（每阶段） | 人 | 见下文 |

五个时机对应关系：**写码时**（L1 实时报错）→ **任务完成时**（L1+L2+L3+手动验收）→ **push 时**（CI 自动跑 L1-L3）→ **切片完成时**（L4+L5）→ **阶段完成时**（L6 用户测试）。

---

## L1 静态检查（已就绪 ✅）

- 命令：`pnpm check`（= typecheck + lint + build）
- 规则：**每个任务提交前必跑**，红了不许提交。这是 AI 写代码时最重要的自动纠错网，能拦住大部分低级错误。
- CI 中自动执行，无需记忆。

## L2 单元测试（待搭建，ROADMAP 0.9）

- 工具：**Vitest**（比 Jest 快、配置少）+ **React Testing Library**（测组件行为）
- 测什么（优先级从高到低）：
  1. **状态机**：Agent 草稿→待审核→已发布→停用的合法/非法流转（AIPaddle 业务逻辑风险最高处）
  2. **权限矩阵**：每个角色能/不能调什么
  3. 工具函数与数据转换
  4. 组件关键交互（表单校验、条件渲染）——只测行为，不测样式
- 约定（已写入 CLAUDE.md）：新功能至少 1 条测试；**修 bug 先写复现测试**（先让测试红，修完变绿，该 bug 永不复发）
- 命令：`pnpm test`（0.9 完成后并入 `pnpm check`）

## L3 集成/API 测试（待搭建，随切片 0 启动）

- 工具：Vitest 直接调用 API 路由 + **Supabase 本地栈**（`supabase start` 在本机 Docker 里起一套完整 Postgres+Auth，测试不碰线上数据）
- 每个 API 必测四件事：
  1. 正常入参 → 正确结果且数据落库
  2. 非法入参 → 4xx 且有错误信息（用 zod 校验）
  3. 无权限角色调用 → 403
  4. **跨租户访问 → 404/403（永远不能返回别家数据）**
- 数据库迁移文件（supabase/migrations/）进 git，保证测试环境与线上结构一致

## L4 E2E 端到端测试（待搭建，ROADMAP 0.9）

- 工具：**Playwright**（真实浏览器自动化）
- 策略：**只测关键路径，不追求覆盖率**。维护一个"冒烟测试集"（Smoke Suite），保持在 10-20 条以内、10 分钟内跑完：
  - 注册 → 登录 → 退出
  - Agent：创建 → 提交审核 → 发布 → 对话 → 日志可见（切片 1 后加入）
  - 知识库：上传 → 提问引用正确（切片 2 后加入）
  - 租户 A 登录看不到租户 B 的任何数据
- 何时跑：每个切片完成时全量跑；CI 中合并到 main 前跑
- **当前原型阶段（mock 数据）的功能与逻辑测试**：后端闭环建立前，原型页面的"测试"= 手动验收 + 少量 Playwright 冒烟（页面能打开、核心交互不报错）。不必为 mock 数据写断言——那些逻辑马上会被真实 API 替换。

## L5 专项测试

**① 租户隔离专项（P0，切片 0 完成时必须有）**
- 固定脚本：创建租户 A/B 各一账号，用 B 的身份枚举尝试访问 A 的每一类资源（API 层逐个打），全部必须 403/404
- RLS 策略测试：直接用 SQL 验证行级安全规则生效
- 任何一条失败 = P0 事故级 bug，停下所有开发先修

**② RAG/AI 质量评测（切片 2 起）**
- 建一个"金标准问答集"（golden set）：≥20 个问题 + 期望答案要点 + 应引用的文档
- 脚本化跑：通过率 ≥80% 且引用正确才算 4.2.3 完成；此后每次改检索/提示词逻辑都重跑防退化
- AI 回答的正确性无法用普通断言测，用"要点命中 + 引用核对"评分

**③ 性能测试（⏸ 后置）**：线上有稳定流量后再做（k6 压测核心 API）。现阶段只守一条：页面首屏 < 3 秒，主观可感知即可。

**④ 安全检查（轻量，持续）**
- `pnpm audit` 查依赖漏洞（CI 中每周跑）
- 密钥不进 git（CI 加 secret 扫描）；`SUPABASE_SERVICE_ROLE_KEY` 只在服务器环境变量

## L6 人工测试

**① 手动验收（每个任务，你来做）**
- 按 Issue 里写的验收标准亲手操作一遍，通过才许关 Issue
- 三不原则：不看代码猜、不信 AI 的"已完成"、不跳步骤

**② 真实用户测试（每个阶段，ROADMAP 阶段 5）**
- 3-5 个目标用户，给任务不给指导（"请创建一个 Agent 并让它回答问题"），你只观察记录
- 每人一页笔记：卡点 / 疑问 / 没用到的功能 / 原话
- 通过标准见 ROADMAP 检测关口 5

## CI：GitHub 提交时的自动测试

工作流文件：`.github/workflows/ci.yml`（已入库）

```
push / PR 到任何分支
  └─ 安装依赖 → typecheck → lint → build        （现在就生效）
  └─ 单元 + 集成测试                              （0.9 完成后启用）
PR 合并到 main 之前
  └─ 以上全部 + Playwright 冒烟集                 （0.9 完成后启用）
merge 到 main 之后
  └─ 自动部署到服务器（ROADMAP 0.8）→ 部署后冒烟（curl 健康检查接口）
```

配套约定：
- GitHub 仓库开启 branch protection：main 分支必须 CI 绿灯才能合并
- CI 红灯 = 最高优先级，先修 CI 再做新任务
- 每个功能走分支 + PR，不直接 push main（哪怕只有你一个人，这让 CI 有拦截点）

## 落实到每个 GitHub Issue 的执行步骤

每个任务 Issue 按 `.github/ISSUE_TEMPLATE/task.md` 创建，执行步骤（也是关闭 Issue 的前置条件）：

1. 读 ROADMAP.md，与用户确认任务编号与验收标准
2. 开发（单次 ≤2h，超了先拆）
3. `pnpm check` 通过（L1）
4. 新增/更新单元或集成测试并通过（L2/L3，纯 UI 样式任务可豁免）
5. 涉及用户路径变化 → 更新并跑 Playwright 冒烟（L4）
6. 涉及租户数据 → 跑租户隔离专项（L5①）
7. 用户手动验收通过（L6①）
8. 分支 push → PR → CI 绿灯 → 合并
9. 更新 ROADMAP.md 状态 → 关闭 Issue

## 命令速查

```bash
pnpm check          # L1：typecheck + lint + build
pnpm test           # L2+L3：单元与集成测试（0.9 后可用）
pnpm test:e2e       # L4：Playwright 冒烟集（0.9 后可用）
supabase start      # 本地起 Supabase 测试栈（切片 0 后可用）
pnpm audit          # 依赖安全检查
```
