## 用例编号规则

`<切片>-<模块>-<序号>`，如 `S0-ISO-01`（切片 0 · 租户隔离 · 第 1 条）、`P-AGT-02`（原型 · Agent 模块 · 第 2 条）。

## 用例字段说明

- **P0/P1/P2**：P0 = 失败即阻塞发布（隔离、安全、数据丢失类）；P1 = 核心功能；P2 = 体验优化
- **层/方式**：对应 TESTING.md 的 L1-L6 层级 + 执行方式（手动 / Vitest / Playwright / 脚本）
- **节点**：对应 ROADMAP 任务编号，即该用例挂到哪个 Issue 下执行

## 与 GitHub Issue 的挂接方式（工作节点执行规则）

1. 每个开发 Issue 创建时，从对应文件中**摘取该任务编号名下的用例 ID 列表**填入 Issue 的「验收标准」栏（如 `[3.2] 注册/登录/退出` → 验收标准写 `S0-AUTH-01 ~ S0-AUTH-06`）。
2. Claude Code 按 TESTING_WORKFLOW.md 协议执行这些用例（可自动化的写成测试代码，手动的逐条操作），结果按用例 ID 逐条回帖。
3. 用例执行状态记号：✅ 通过 / ❌ 失败 / ⏭ 豁免（须注明原因）/ 🔁 修复后复测通过。
4. 新发现的缺陷场景 → 补写成新用例（编号顺延）→ 提交进本用例库，防止回归。
