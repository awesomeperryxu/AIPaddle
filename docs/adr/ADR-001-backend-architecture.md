# ADR-001：后端架构选型

- 状态：**已采纳**（2026-07-18，Perry 拍板）
- 对应任务：ROADMAP 2.1（并部分覆盖 2.2 认证、2.6 轮子清单）
- 决策人：Perry
- 起草：Claude

## 决策

**后端形态**：不建独立后端服务，后端逻辑全部写在 Next.js 内（API Routes / Server Actions），前后端一个仓库、一次部署——即"全用 Next.js 一个框架搞定前后端"的单体架构。

**数据库与基础服务**：使用 **Supabase 云托管**，一站式提供：

| 能力 | Supabase 模块 | 替代了什么 |
|------|--------------|-----------|
| 数据库 | Postgres（标准版） | 自建/云托管 PG |
| 认证 | Supabase Auth | 自写登录、Auth.js 等 |
| 文件存储 | Supabase Storage | 自建对象存储（知识库文档上传用） |
| 向量检索 | pgvector 扩展 | 独立向量数据库（RAG 用） |
| 管理后台 | Supabase Dashboard | 自写数据库管理界面 |

## 理由

1. **运维成本最低**：一个人开发，不该把时间花在数据库备份、高可用、安全补丁上。
2. **认证不自己写**：登录/注册/找回密码是最不该自研的模块，Supabase Auth 开箱即用。
3. **与技术栈同源**：Supabase 官方对 Next.js 支持最好，AI（Claude Code/v0）对这套组合训练数据最充分，生成代码质量高。
4. **无锁定风险**：底层是标准 Postgres，未来如需迁回国内（自建或腾讯云托管 PG），数据可整体导出迁移，应用层改动可控。

## 已知风险与缓解措施

**主要风险：Supabase 服务器在海外，国内访问延迟高且可能不稳定。**

缓解措施（写入开发约定，必须遵守）：

1. **浏览器永不直连 Supabase**。所有前端请求只打到自己的 Next.js API（部署在腾讯云服务器 43.173.99.218），由服务端再访问 Supabase。用户感知的稳定性取决于你的服务器，而不是用户到 Supabase 的线路。
2. **Supabase 项目区域选离中国近的**（新加坡或东京），降低服务器 → Supabase 的延迟。
3. **敏感配置只放服务端**：`SUPABASE_SERVICE_ROLE_KEY` 等密钥只存在于服务器环境变量，绝不进前端代码或 git。
4. **退路预留**：如实测延迟不可接受，迁移路径为"同构迁移"——导出 Postgres 数据到自建/腾讯云 PG，认证换 Better Auth；因为第 1 条约定隔离了直连，前端代码几乎不用改。

## 多租户隔离方案（原则，细化在任务 2.2/2.4）

- 每张业务表带 `org_id` 字段；
- 双层防护：数据库层用 Postgres RLS（行级安全）按 `org_id` 隔离 + 应用层 API 中间件校验，单层失误不至于泄数据；
- 跨租户数据可见即 P0 事故，切片 0 完成时必须有专门的隔离测试（ROADMAP 3.3）。

## 被否决的备选方案

| 方案 | 否决原因 |
|------|---------|
| 自建 Postgres（Docker @ 腾讯云服务器） | 数据在国内、零月费，但备份/运维/认证全要自己搞，对单人团队负担重（Perry 已知晓权衡后选择 Supabase） |
| 腾讯云托管 PostgreSQL | 国内访问快、免运维，但仍需自研认证与存储，且有月费 |
| 独立后端服务（NestJS/FastAPI） | 两个代码库、两套部署，复杂度翻倍，单人团队无收益 |

## 连带决定

- ORM/数据访问：服务端统一用 `@supabase/supabase-js` + 数据库迁移用 Supabase CLI（migration 文件进 git）；
- 该决策落地后，ROADMAP 2.6「不自己造的轮子」清单中"认证、文件存储"两项视为已定。

## 复审条件

出现以下任一情况时重开本决策：服务器 → Supabase 实测延迟中位数 > 500ms；Supabase 国内可达性持续恶化；获得需要"数据必须存境内"的企业客户。
